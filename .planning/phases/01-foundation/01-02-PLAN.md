**File 1 â€” `src/server/db/client.ts`:**
```typescript
import { neon } from '@neondatabase/serverless';
import { drizzle } from 'drizzle-orm/neon-http';
import { sql } from 'drizzle-orm';
import * as schema from './schema';
import { env } from '@/lib/env';

const queryClient = neon(env.DATABASE_URL);
export const db = drizzle(queryClient, { schema });

// THE ONLY approved path to run queries on tenant-scoped tables (Pitfall 1).
// Sets the `app.tenant_id` Postgres session variable that RLS policies reference.
export async function withTenant<T>(
  tenantId: string,
  fn: (tx: typeof db) => Promise<T>
): Promise<T> {
  // Note: neon-http driver does NOT support multi-statement transactions in the
  // same way the ws driver does. For neon-http we set the GUC inline per query
  // via a wrapping transaction using db.transaction (supported since drizzle 0.45):
  return db.transaction(async (tx) => {
    await tx.execute(sql`SELECT set_config('app.tenant_id', ${tenantId}, true)`);
    return fn(tx as unknown as typeof db);
  });
}
```

If `drizzle-orm/neon-http` `db.transaction` is not supported at runtime (neon-http is known to have limitations here), fall back to this variant that sets the GUC and runs `fn` without a transaction wrapper â€” RLS still applies:

import { withTenant } from '@/server/db/client';

import { getTenantId } from '@/lib/tenant-context';

  return tenant;
});

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| application code â†’ Postgres | All tenant-scoped queries MUST pass through `withTenant()`. Raw `db.select()` on a tenant table bypasses the GUC and â€” if RLS is not yet installed â€” returns cross-tenant rows |
| client â†’ headers() | `x-tenant-id` header is set by Vercel-managed edge middleware and is trusted only in server code. Vercel strips external `x-tenant-id` headers from incoming requests before they reach middleware, but defense-in-depth: `getTenantId()` never reads from query/body |
| migration script â†’ live DB | drizzle-kit push runs against production schema. A typo in the schema file that silently drops a UNIQUE constraint is a catastrophic silo regression |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Information Disclosure | Cross-tenant data leak via query missing tenant_id filter (Pitfall 1) | mitigate | `withTenant()` is the sole approved DB entry point. Task 2 enforces the pattern in code; future RLS policies (post-Phase 1) will be the defense-in-depth layer. Integration test in Task 3 verifies the happy path. |
| T-02-02 | Elevation of Privilege | UNIQUE(email) instead of UNIQUE(tenant_id, email) (Pitfall 2) | mitigate | Schema enforces `uniqueIndex('users_tenant_id_email_unique').on(t.tenantId, t.email)`.
