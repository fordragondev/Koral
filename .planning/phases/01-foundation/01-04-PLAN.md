**Step 3 — Create `src/server/auth/config.ts`** using the EXACT pattern from 01-PATTERNS.md `src/server/auth/config.ts` section. Key differences from the pattern to honor:

```typescript
// server/auth/config.ts
import NextAuth from 'next-auth';
import { DrizzleAdapter } from '@auth/drizzle-adapter';
import Google from 'next-auth/providers/google';
import Apple from 'next-auth/providers/apple';
import Email from 'next-auth/providers/nodemailer';  // v5 renamed Email → Nodemailer
import { db } from '@/server/db/client';
import { env } from '@/lib/env';
import { sendOtpEmail } from './email-sender';

export const { handlers, signIn, signOut, auth } = NextAuth((req) => {
  // Derive tenantId from middleware header — NEVER from client-supplied input (D-11, Pitfall 5)
  const tenantId = req?.headers.get('x-tenant-id') ?? '';
  // Derive cookie domain from exact request host — NEVER a wildcard parent (Pitfall 3)
  const host = req?.headers.get('host')?.replace(/:\d+$/, '') ?? undefined;

  return {
    adapter: DrizzleAdapter(db),
    secret: env.NEXTAUTH_SECRET,
    session: { strategy: 'database' },  // AUTH-03: DB sessions persist across refresh
    providers: [
      Email({
        // 6-digit numeric OTP per D-08 — NOT a UUID magic link
        generateVerificationToken: () =>
          Math.floor(100000 + Math.random() * 900000).toString(),
        sendVerificationRequest: async ({ identifier, token }) => {
          await sendOtpEmail(identifier, token);
        },
        // Short-lived OTP (Claude's Discretion per CONTEXT.md) — 10 minutes
        maxAge: 10 * 60,
      }),
      Google({
        clientId: env.GOOGLE_CLIENT_ID,
        clientSecret: env.GOOGLE_CLIENT_SECRET,
      }),
      ...(env.APPLE_CLIENT_ID && env.APPLE_CLIENT_SECRET
        ? [Apple({ clientId: env.APPLE_CLIENT_ID, clientSecret: env.APPLE_CLIENT_SECRET })]
        : []),
    ],
    callbacks: {
      async session({ session }) {
        // D-11: every session payload carries tenantId
        (session as any).tenantId = tenantId;
        return session;
      },
      async jwt({ token }) {
        (token as any).tenantId = tenantId;
        return token;
      },
    },
    cookies: {
      sessionToken: {
        name: `__Secure-koral.session-token-${tenantId || 'unknown'}`,
        options: {
          // Pitfall 3: exact tenant host ONLY. No wildcard parent domains.
          domain: host,
          httpOnly: true,
          sameSite: 'lax',
          secure: process.env.NODE_ENV === 'production',
          path: '/',
        },
      },
    },
  };
});
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → `/api/auth/*` route handler | All auth requests (OTP send, OTP verify, OAuth callback, signOut) cross this boundary. Auth.js owns CSRF, rate limiting, and nonce handling |
| Request headers → `tenantId` derivation | `x-tenant-id` is stamped by Plan 03 middleware. Auth config trusts it — derived client headers are ignored |
| Auth.js → DrizzleAdapter → DB | Adapter runs as the auth layer. Must respect the Plan 02 UNIQUE(tenant_id, email) constraint — if the adapter sends an insert without tenant_id, it fails at the DB. That's the correct failure mode |
| Session cookie → browser → next request | Cookie is bound to exact host via `cookies.sessionToken.options.domain` — a cookie set on aquariumcommu.com is NOT sent to anime.test.koral.local |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Spoofing | Client-supplied `tenantId` in body/query bypasses silo | mitigate | Config derives tenantId ONLY from `req?.headers.get('x-tenant-id')`. Grep test `tests/unit/auth.test.ts` asserts this. |
| T-04-02 | Information Disclosure | Session cookie bleeds across tenants via wildcard domain (Pitfall 3) | mitigate | `cookies.sessionToken.options.domain` is set to the exact request host. Grep test asserts no `domain: '.` string appears. Plan 06 adds an E2E test that authenticates on tenant A and asserts no session exists on tenant B. |
| T-04-03 | Spoofing | Cross-tenant OTP code reuse (code generated for tenant A works on tenant B) | mitigate | Plan 02 schema: `verificationTokens` has `UNIQUE(tenant_id, identifier, token)`. Auth.js DrizzleAdapter inserts include tenant_id because the users table requires it. An OTP for (aquariumcommu.com, alice@example.com) fails lookup on (anime.test, alice@example.com). |
| T-04-04 | Elevation of Privilege | Brute force on 6-digit OTP (1M combinations) | mitigate | Auth.js enforces `maxAge: 600` seconds (10-min window). Rate limiting on `/api/auth/signin/email` is accepted as a deferred hardening item — Phase 1 ships with Auth.js's built-in protection; Phase 4 (safety gate) adds Upstash `@upstash/ratelimit` per RESEARCH.md Supporting Libraries. Documented as an accepted gap. |
| T-04-05 | Replay | OTP token reuse after successful verification | mitigate | Auth.js Email provider deletes verification tokens on successful verification. Standard library behavior — no custom code required. |
| T-04-06 | Spoofing | OAuth `state` parameter forgery | mitigate | Auth.js v5 enforces OAuth state + PKCE by default. No custom handling required. |
| T-04-07 | Tampering | v4 route handler pattern (`export default NextAuth(...)`) silently 404s on App Router (Pitfall 8) | mitigate | Grep test asserts `export const { GET, POST } = handlers` and asserts `export default NextAuth` does NOT appear. |
| T-04-08 | Denial of Service | OTP send endpoint abuse (send 10k codes to one address) | accept | Deferred to Phase 4. Phase 1 logs OTPs to console in dev, so there is no real send surface yet. |
</threat_model>
